# Инвентаризация проекта `set10-payment-plugin`

Дата фиксации: 2026-04-08

## 1) Краткая сводка

- Тип проекта: платежный плагин Set Retail 10 (Java, Maven).
- Цель: интеграция кассы с SBG Pay POS API для QR-оплаты.
- Текущее состояние: рабочая сборка есть, ключевой payment-flow реализован, но есть функциональные и инженерные пробелы.

## 1.1) Прогресс после инвентаризации (2026-04-08)

- Добавлен fallback для QR: теперь учитываются данные из `create payment` ответа и поля `qrCodeData` в status-ответе.
- Исправлена логика таймаутов polling: timeout HTTP-запроса статуса теперь ограничивается оставшимся временем операции.
- Реализован refund-flow с конфигурируемыми endpoint'ами:
  - запуск возврата через `sbgpay.refundCreateUrl`;
  - polling возврата через `sbgpay.refundStatusUrl` (для промежуточных статусов);
  - добавлены persisted-поля возврата.
- Удалены неиспользуемые элементы:
  - `sbgpay.qrUrlTemplate` в конфигурации/metainf;
  - неиспользуемые UI-ключи подтверждения оплаты.
- Исправлена инструкция в README по имени артефакта (`SBGPayPaymentPlugin-2.0.0.jar`).
- Добавлен базовый CI workflow (`.github/workflows/build.yml`) для проверки сборки.

## 2) Что уже сделано (подтверждено)

- Реализован основной flow оплаты:
  - загрузка методов оплаты;
  - выбор метода;
  - ввод суммы (включая частичную оплату);
  - создание платежа;
  - polling статуса;
  - завершение оплаты на кассе;
  - отправка `/complete` после фискализации;
  - отправка `/cancel` при отмене операции.
- Реализовано сохранение полей платежа в чек:
  - `sbgpay.paymentId`, `sbgpay.paymentCode`, `sbgpay.methodId`, `sbgpay.methodName`, `sbgpay.status`.
- Реализована локализация UI-строк:
  - `ru`, `en`.
- Есть сборка JAR и fat-JAR через Maven.
- Базовые таймауты и обработка HTTP-ошибок присутствуют.

## 3) Что не сделано (подтверждено)

- `Refund` реализован частично:
  - логика возврата есть, но требует подтвержденного API-контракта и настройки endpoint'ов (`sbgpay.refundCreateUrl`, `sbgpay.refundStatusUrl`);
  - нужна проверка на интеграционном стенде с реальным SBG API.
- Тестовый контур частичный:
  - добавлены базовые unit-тесты (QR fallback и статусные предикаты);
  - нет интеграционных тестов с моками API;
  - нет smoke-тестов release-артефакта.
- Нет полноценного CI/CD:
  - добавлен базовый pipeline сборки, но пока нет quality gates, тестового контура и release-процесса.
- Есть технический долг по сборке:
  - `systemPath`-зависимость на `lib/set10pos-api-1.20.0.jar`.
- По-прежнему есть потенциальный техдолг в структуре основного класса (крупный монолитный `SbgPayPaymentPlugin.java`).

## 5) Отдельно: проработка возврата (Refund)

Текущее состояние:

- Подтверждено: возврат не поддерживается в runtime и блокируется на уровне UI.

Связанный документ:

- `REFUND_IMPLEMENTATION_PLAN.md` — детальный план реализации, этапы, риски, DoD.

Нужно проверить перед реализацией:

- Какие операции поддерживает SBG Pay API для возвратов:
  - полный/частичный возврат;
  - ограничения по времени;
  - требования к идентификаторам исходного платежа.
- Какой ожидается процесс в Set Retail 10:
  - события плагина при возврате;
  - связь с чеком возврата и фискализацией.

Целевая реализация возврата (этапы):

1. Аналитика контракта:
   - подтвердить API-эндпоинты и поля;
   - согласовать бизнес-правила и ограничения.
2. Архитектура:
   - определить state-machine возврата;
   - определить persisted-поля для refund (`refundId`, `refundStatus`, `sourcePaymentId` и т.д.).
3. Реализация:
   - заменить заглушку `doRefund()` на реальный flow;
   - добавить polling/timeout/error handling для refund;
   - корректно показать статусы кассиру и покупателю.
4. Тестирование:
   - позитивные, негативные и edge-кейсы;
   - частичный возврат;
   - retry/idempotency.
5. Ввод в эксплуатацию:
   - пилот на ограниченном контуре;
   - мониторинг ошибок и rollback-plan.

Критерии готовности возврата (DoD):

- Возврат проходит end-to-end на стенде и в пилоте.
- Поддержаны минимум полный возврат и отказоустойчивый retry.
- Ошибки API корректно отображаются кассиру.
- Есть автотесты и инструкции эксплуатации.

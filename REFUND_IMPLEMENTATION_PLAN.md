# План реализации возврата (Refund) для SBG Pay плагина Set10

Дата: 2026-04-08
Статус: Draft (для согласования и запуска разработки)

## 1) Цель

Реализовать возврат оплаты через SBG Pay в Set Retail 10 с учетом:

- корректного поведения кассы и UI;
- фискализации чека возврата;
- отказоустойчивости (retry/idempotency);
- прозрачной диагностики ошибок для кассира и поддержки.

## 2) Текущее состояние (подтверждено)

- `doRefund()` сейчас не выполняет возврат и завершает операцию как `paymentNotCompleted`.
- В проекте нет автотестов и CI.
- Плагин уже умеет:
  - создавать оплату;
  - отслеживать статус;
  - делать `/complete` после фискализации;
  - делать `/cancel` при отмене.

## 3) Что нужно подтвердить до кодинга (внешний контракт)

Нужно получить подтверждение от API-команды SBG Pay по возврату:

1. Эндпоинты refund:
   - URL;
   - HTTP-метод;
   - схема запросов/ответов;
   - статус-коды.
2. Бизнес-правила:
   - поддержка полного/частичного возврата;
   - ограничения по сроку (T+N дней/часов);
   - можно ли делать несколько частичных возвратов.
3. Идентификаторы:
   - что является ссылкой на исходную оплату (`paymentId`, `paymentCode`, внешний `order.id`).
4. Идемпотентность:
   - обязательность `Idempotency-Key`;
   - TTL ключа;
   - поведение при повторе.
5. Финализация:
   - нужно ли отдельное `/refunds/{id}/complete` после фискализации возврата;
   - какие статусы считаются финальными.

## 4) Варианты реализации

### Вариант A (быстрый результат)

Реализовать только полный возврат без частичных сценариев.

Плюсы:

- быстрый вывод в эксплуатацию;
- минимальный объем изменений;
- ниже риск регрессий.

Минусы:

- не покрывает частичные возвраты;
- возможна повторная переделка архитектуры.

Риск:

- средний (бизнес-ограничение на стороне кассы/оператора).

### Вариант B (правильная база на будущее)

Реализовать полный и частичный возврат с полноценной state-machine.

Плюсы:

- закрывает все основные бизнес-сценарии;
- меньше технического долга;
- лучше для масштабирования.

Минусы:

- дольше по срокам;
- выше объем тестирования и интеграционной проверки.

Риск:

- выше на этапе внедрения, ниже в долгую.

Рекомендация:

- Если нужен быстрый запуск: стартовать с Варианта A, но с архитектурными точками расширения под B.
- Если есть окно на качественный релиз: сразу Вариант B.

## 5) Целевая архитектура Refund (минимум изменений в текущем коде)

## 5.1 Поток возврата

1. Кассир инициирует возврат.
2. Плагин валидирует наличие `sourcePaymentId` (из данных исходной оплаты).
3. Плагин отправляет запрос на refund в SBG API.
4. При async-модели:
   - запускается polling статуса refund.
5. При успешном статусе:
   - плагин завершает возврат в Set10 через callback.
6. После фискализации чека возврата:
   - при необходимости отправляется confirm/complete refund.

## 5.2 Persisted поля (рекомендуется добавить)

- `sbgpay.refundId`
- `sbgpay.refundCode`
- `sbgpay.refundStatus`
- `sbgpay.sourcePaymentId`
- `sbgpay.refundError` (опционально, только для диагностики)

Примечание:

- ключи нужно добавить в `metainf.xml` и `strings_ru.xml`/`strings_en.xml`.

## 5.3 Обработка ошибок

- Ошибки валидации (нет исходной оплаты, превышен лимит суммы, срок возврата истек).
- Технические ошибки API (timeout/5xx/429).
- Ошибки авторизации (401).
- Сценарий retry при временных ошибках.

Требование:

- UI должен показывать человеку короткий и понятный текст;
- подробности ошибок уходят в логи.

## 5.4 Идемпотентность и повторы

- Все `POST` на refund только с `Idempotency-Key`.
- При сетевой ошибке использовать повтор с тем же ключом в рамках одного бизнес-действия.
- Логи должны связывать попытки по `refundId` и `sourcePaymentId`.

## 5.5 Таймауты и polling

- отдельные настройки для refund:
  - `sbgpay.refundPollDelayMs`
  - `sbgpay.refundPollTimeoutSeconds`

Если отдельные настройки не вводить:

- использовать текущие `pollDelayMs`/`pollTimeoutSeconds` как fallback.

## 8) Критерии готовности (DoD)

- Возврат выполняется end-to-end на стенде.
- Ошибки корректно отображаются кассиру и логируются.
- Сохранены refund-поля для аналитики и выгрузки.
- Нет регрессий основного payment flow.
- README и метаданные синхронизированы с кодом.

## 9) Основные риски и меры

1. Неоднозначный API-контракт refund.
   - Мера: сначала фиксируем контракт письменно.
2. Регрессии в оплате из-за изменений в общем классе.
   - Мера: выделить минимальные точки изменения, добавить smoke-сценарии.
3. Неочевидные бизнес-ограничения на частичный возврат.
   - Мера: запуск через feature flag по этапам.
